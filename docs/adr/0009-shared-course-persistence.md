# 0009. 공유 코스 영속성 — URL 인코딩 vs 서버 저장

- 상태: Accepted
- 제안일: 2026-05-30
- 결정일: 2026-05-30
- 결정자: 프런트엔드 팀

> 출처: [PRD §10 OQ5](../PRD.md#10-가정-및-미결-open-questions).

## 맥락

- 제안서([PRD §4 F3](../PRD.md#f3-one-click-share--카카오톡-기반-비용-분담-공유))는 완성 코스 + 상세 예산 내역(항목별 금액·1인당 송금액)을 카카오톡/링크로 공유한다.
- 공유 링크를 받은 동행인이 코스 데이터를 어디서 조회하는지 — **저장 위치**가 미정이다.

## 고려한 대안

- **A: URL 인코딩(쿼리/해시에 코스 직렬화)** — 서버 불필요, 즉시 구현 가능. 단 URL 길이 제한, 데이터 노출, 추후 편집·조회 통계 불가.
- **B: 서버 저장 후 단축 ID 링크(`/share/:id`)** — 영속·조회·통계·편집 추적 가능. 백엔드 필요 → **[0007](./0007-backend-db-ownership.md)** 결정에 의존.
- **C: 압축 인코딩 + 선택적 서버 백업** — 가벼운 공유는 URL, 영속 필요 시 서버. 구현 복잡도 증가.

## 결정

**단계적 채택: 임시 A(URL 인코딩) → 목표 B(서버 저장 + 단축 ID).**

- **현재(백엔드 연동 전, UI-우선 단계)**: **A — URL 인코딩**을 채택한다. 코스+예산을 해시/쿼리에 직렬화하고, 공유 수신 라우트(`/share/:id` 또는 `/share#<payload>`)가 이를 디코딩해 렌더한다. 백엔드 없이 공유가 실제 동작한다.
- **목표(백엔드 계약 확정 후)**: **B — 서버 저장 + 단축 ID**로 전환한다. [0007](./0007-backend-db-ownership.md)이 `Accepted`(별도 백엔드 존재)이므로 영속·조회 통계·편집 추적이 가능하다.
- 직렬화 스키마는 두 단계에서 동일하게 쓸 수 있도록 **버전 필드(`v`)** 를 포함해 설계한다(URL payload ↔ 서버 저장 레코드 호환).

## 결과

긍정적:
- 백엔드 미완 상태에서도 공유 기능을 데모·검증할 수 있다.
- 스키마에 버전 필드를 두어 A→B 전환 시 동일 직렬화 포맷을 재사용한다.

부정적/후속:
- A 단계 한계: URL 길이 제한(코스가 길면 압축 필요), 데이터가 링크에 노출, 조회 통계·사후 편집 불가. → B 전환으로 해소.
- 결정 후: 코스·예산 직렬화 스키마 확정, 공유 라우트 추가, 카카오톡 공유하기 API 메시지 템플릿(요약 카드) 연계.
- B 전환 시 필요한 서버 측 공유 API(생성/조회 엔드포인트·단축 ID 규칙)는 백엔드와 협의한다. → [백엔드 협의 문서](../backend-api-contract.md)
