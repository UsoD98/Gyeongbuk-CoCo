# 0007. 백엔드/DB 소유 주체와 관광 데이터 수집 위치

- 상태: Accepted
- 결정일: 2026-05-30
- 결정자: 프런트엔드 팀

> 출처: [PRD §10 OQ1](../PRD.md#10-가정-및-미결-open-questions).

## 맥락

- 제안서([PRD §6](../PRD.md#6-데이터-활용-한국관광공사-openapi--필수))는 **"관광공사 OpenAPI 데이터를 서비스 DB에 적재하고 월 1회 주기로 수집"**, **"음식점 평균 식비·숙박 요금 분석 후 메타데이터 생성"**, **"MapX/MapY로 동선·교통비 계산"** 을 전제한다.
- 그러나 본 repo는 **프런트엔드 전용**(React SPA)이며 백엔드·DB가 없다.
- 풀어야 할 제약: 관광공사 API 키 노출 방지, CORS, rate limit, 배치 수집(월 1회), 가공 로직(인원 버킷 분류·예산 근사치 산출)의 실행 위치.

## 고려한 대안

- **A: 프런트에서 OpenAPI 직접 호출** — 가장 빠른 착수. 단, API 키가 클라이언트에 노출되고 CORS·rate limit에 취약하며, 월 1회 배치 수집·가공이 불가능(제안서 전제와 충돌).
- **B: 별도 백엔드(API 서버 + DB)** — 제안서 전제와 정합. 키 보호, 배치 수집, 메타데이터 가공을 서버가 담당. 운영 비용·개발 범위 증가.
- **C: 서버리스 함수 + 관리형 DB(프록시·캐시 중심)** — 중간안. 키 프록시·응답 캐시는 해결하나 본격 배치/가공 파이프라인엔 부족할 수 있음.

## 결정

**B(별도 백엔드) 채택.** 단, 백엔드·DB·수집 배치는 **별도 repo/팀**이 담당하며 **이 repo는 프런트엔드 전용**이다.

- 관광공사 OpenAPI 호출, DB 적재, 월 1회 수집, 음식점·숙박 메타데이터 가공, MapX/Y 기반 동선·교통비 계산은 **모두 백엔드 책임**.
- 이 repo의 `src/api/`는 관광공사 API를 직접 호출하지 않고 **자체 백엔드 API만 소비**한다.

## 결과

긍정적:
- 관광공사 API 키가 클라이언트에 노출되지 않는다(백엔드 시크릿). CORS·rate limit·배치 수집을 백엔드가 흡수.
- 인원 버킷 분류·예산 근사치 같은 가공 로직을 FE가 떠안지 않는다.

부정적/후속:
- FE는 백엔드 API 계약(엔드포인트·응답 스키마)에 의존 → 계약 확정 전 `src/api/` 본구현 불가. 그 전까지는 타입·목 데이터로 진행.
- FE 환경변수는 백엔드 base URL 정도(`VITE_API_BASE_URL` 등). 관광공사 키는 FE에 두지 않는다.
- **[0009](./0009-shared-course-persistence.md)**(공유 코스 영속성)의 전제가 풀렸다 — 백엔드가 있으므로 서버 저장(B) 선택지가 유효.
