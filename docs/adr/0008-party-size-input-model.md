# 0008. 인원 입력 모델 — 버킷 vs 자유 카운터

- 상태: Accepted
- 제안일: 2026-05-30
- 결정일: 2026-05-30
- 결정자: 프런트엔드 팀

> 출처: [PRD §10 OQ2](../PRD.md#10-가정-및-미결-open-questions).

## 맥락

- 제안서([PRD §4 F1](../PRD.md#f1-who--how-many--인원수-기반-스마트-큐레이션))는 **1인 / 2인 / 3~4인 버킷** 기준 큐레이션을 핵심 차별점으로 둔다(혼밥 바테이블·감성 숙소·패밀리룸).
- 그러나 현재 `src/pages/Index.tsx`의 인원 입력은 **자유 증감 카운터**(`number`, 하한 1, 상한 없음)다.
- 두 모델이 어긋나 있어 추천·예산 로직 착수 전 통일이 필요하다.

## 고려한 대안

- **A: 버킷 셀렉트로 교체(1 / 2 / 3~4)** — 제안서·추천 로직과 정확히 정합, 데이터 분류와 1:1. 5인 이상 등 자유 인원 표현은 포기.
- **B: 카운터 유지 + 내부 매핑(n → 버킷)** — 입력 유연. 단 UI(임의 숫자)와 추천(버킷) 사이 괴리, 5인+ 처리 모호.
- **C: 하이브리드** — 카운터로 받되 추천은 버킷으로 매핑, 5인+는 `3~4` 버킷으로 취급하고 안내 문구 표시.

## 결정

**B(카운터 유지 + 내부 매핑) 채택.**

- 입력 UI는 현재의 자유 증감 카운터(`number`, 하한 1)를 **그대로 유지**한다.
- **추천 쿼리**는 카운터 값 `n`을 버킷으로 매핑해 전달한다.
  - `n === 1` → `1`
  - `n === 2` → `2`
  - `n >= 3` → `'3-4'` (5인 이상도 `'3-4'` 버킷으로 취급하고 안내 문구 표시)
- **예산 1인당 분배**는 버킷이 아니라 **실제 카운터 값 `n`** 으로 나눈다(총예산 ÷ n). 추천은 버킷, 분배는 실인원이라는 점을 분리한다.

매핑은 단일 헬퍼(예: `toPartyBucket(n)`)로 캡슐화해 검색조건 store에서 일관 사용한다.

## 결과

긍정적:
- 입력 유연성 유지(임의 인원 표현 가능), 1인당 부담액이 실제 동행 인원과 일치해 더치페이 직관과 부합.

부정적/후속:
- UI(임의 숫자)와 추천(버킷) 사이 표현 괴리가 존재 → 5인+ 안내 문구로 보완.
- 검색 파라미터 타입: 카운터 `partySize: number`(원본) + 매핑 결과 `partyBucket: 1 | 2 | '3-4'`(추천 전송용)을 함께 둔다.
- 숙박 분류(1·2·4인)와의 매핑 규칙은 백엔드 추천 계약 확정 시 함께 맞춘다([PRD §6 ②](../PRD.md#6-데이터-활용-한국관광공사-openapi--필수)). → [백엔드 협의 문서](../backend-api-contract.md)
- 이 매핑·분배 규칙은 [실시간 예산 모델](../PRD.md#f2-smart-itinerary--budget-planner--실시간-예산-계산기--코스-빌더)에 직접 반영된다.
