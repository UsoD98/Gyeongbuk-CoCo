# 0003. 전역 상태 관리에 Zustand 채택

- 상태: Accepted
- 결정일: 2026-05-05
- 결정자: 프런트엔드 팀

## 맥락

- 우선 필요한 전역 상태는 **테마**(light/dark) 정도로 작다.
- 곧 인증 사용자·필터·즐겨찾기 같은 상태가 더해질 전망.
- 보일러플레이트가 적어야 하고, 컴포넌트 외부(예: 모듈 로드 시점)에서도 상태에 접근할 수 있어야 한다 — 새로고침 시 테마 깜빡임을 막기 위해.

## 고려한 대안

- **Redux Toolkit** — 표준이지만 보일러플레이트가 많고 비동기 흐름이 과함. 본 프로젝트 규모에 비해 무겁다.
- **Recoil** — Facebook 유지보수 종료. 후보에서 제외.
- **Jotai** — atom 단위가 가볍지만, "모듈 로드 시점에 DOM 갱신" 같은 부수효과 패턴이 자연스럽지 않다.
- **Context + useReducer** — 의존성 0이지만 store 분리·셀렉터·외부 접근이 직접 만들기에 번거롭다.
- **Zustand 5** — 훅 하나로 store 정의, `getState()`로 컴포넌트 외부 접근 가능, 미들웨어(persist 등)도 옵션.

## 결정

**Zustand 5** 채택. 컨벤션:

- 도메인별 `src/stores/<name>Store.ts`, 훅 이름 `use<Domain>Store`.
- 영속화(`localStorage`)와 DOM 부수효과는 **setter 내부**에 둔다.
- 초기 깜빡임 방지가 필요하면 파일 하단에 *모듈 로드 시점* 초기화 한 줄.

## 결과

긍정적:
- `themeStore`가 30줄 안에 영속화 + DOM 갱신 + 토글 액션을 모두 담는다.
- 컴포넌트에서는 `useThemeStore()` 한 줄로 끝난다.
- 새 도메인 추가 시 같은 패턴을 복제하면 된다.

부정적:
- DevTools·middleware 활용은 별도 도입 시점에 ADR 갱신 필요.
- 큰 비즈니스 상태(서버 캐시 등)가 생기면 TanStack Query 같은 별도 도구와 역할 분리를 다시 결정해야 한다.

후속:
- 사용 패턴은 [CONVENTION.md §6](../../CONVENTION.md)에 정리.
