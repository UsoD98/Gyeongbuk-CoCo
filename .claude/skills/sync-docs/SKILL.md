경북 CoCo 저장소의 코드 변경을 읽고, **이미 저장소에 문서화된 규칙**에 따라 관련 문서를 갱신한다.
새 규칙을 발명하지 말 것 — CONTRIBUTING.md·docs/adr/README.md·CHANGELOG.md에 적힌 규칙을 *실행*하는 스킬이다.

## 1. 변경 수집

먼저 무엇이 바뀌었는지 확정한다.

```powershell
git diff --cached --name-only        # staged 파일
git diff --cached --stat             # 변경 규모
```

staged가 비어 있으면 직전 커밋(`git show --stat HEAD`)이나 작업 중인 변경(`git diff --name-only`)을 대상으로 삼고, 무엇을 기준으로 분석하는지 사용자에게 한 줄로 알린다.

커밋 메시지(있다면)의 `<type>(<scope>)`도 신호로 쓴다. 없으면 변경 내용에서 type을 추론한다.

## 2. 매핑 규칙 적용

아래 표대로 각 대상 문서의 필요 여부를 판정한다. **저위험은 자동 적용, 고위험은 초안을 제시하고 승인받는다.**

| 변경 성격 | 대상 문서 | 강도 |
| --- | --- | --- |
| 사용자가 인지하는 기능/버그/동작 변경 (`feat`/`fix`/`Changed`/`Removed`/`Security`) | `CHANGELOG.md` `[Unreleased]` | **자동 적용** |
| 외부 의존성 도입·교체, 폴더 구조·레이어·라우팅·상태관리 등 광역 패턴, 되돌리기 어려운 결정(데이터 모델·인증 흐름·빌드 출력) | `docs/adr/NNNN-*.md` 신규 | **승인형 (인터뷰)** |
| 신규 코딩 패턴·규칙 | `CONVENTION.md` | **승인형** |
| 신규 디자인 토큰·색·타이포·컴포넌트 패턴 | `DESIGN.md` | **승인형** |
| 디렉터리 구조·npm 명령어·스택 변경 | `CLAUDE.md` / `README.md` | **승인형** |

판정이 모두 "불필요"면 그 사실을 사용자에게 알리고 종료한다. 억지로 문서를 만들지 않는다.

## 3. CHANGELOG (자동 적용)

- `[Unreleased]` 섹션의 알맞은 카테고리(`Added`/`Changed`/`Deprecated`/`Removed`/`Fixed`/`Security`)에 **한 줄** 추가.
- 문구는 커밋 메시지 설명을 사용자 관점으로 다듬어 작성. *무엇을*이 아니라 *사용자에게 무엇이 달라지는지*.
- 카테고리 자리표시자("작업 중인 변경은…") 줄은 실제 항목이 생기면 지운다.
- 해당 카테고리가 없으면 `### Added` 등 헤더를 만들어 추가. 형식은 Keep a Changelog 1.1.0.

## 4. ADR (승인형 + 인터뷰)

새 ADR이 필요하다고 판정되면:

1. 다음 일련번호를 계산: `docs/adr/` 의 `NNNN-*.md` 중 최대값 + 1, 4자리 zero-pad.
2. diff·코드로 알 수 있는 부분(맥락 일부·결정·결과)을 먼저 채운다.
3. **diff만으로 알 수 없는 항목은 사용자에게 1~2개만 질문한다:**
   - "이 [의존성/패턴]을 왜 골랐나요?"
   - "어떤 대안을 검토하고 버렸나요?"
   - "감수한 트레이드오프는?"
4. 사용자가 답하면 본문에 녹인다. **"나중에"라고 하면 해당 항목을 `> TODO: <무엇> 작성` 마커로 남기고 진행** — 흐름을 막지 않는다. 추측으로 '고려한 대안'을 지어내지 말 것(틀린 근거 박제 방지).
5. `docs/adr/README.md`의 **목록 표에 행을 추가**하고 상태는 보통 `Accepted`(또는 `Proposed`).

ADR 템플릿(docs/adr/README.md 기준):

```markdown
# NNNN. <짧은 제목>

- 상태: Accepted
- 결정일: <오늘 YYYY-MM-DD>
- 결정자: <사용자에게 확인 — 기본 "프런트엔드 팀">

## 맥락
## 고려한 대안
## 결정
## 결과
긍정적:
부정적:
후속:
```

결정일은 커밋/작업 시점의 실제 날짜를 사용한다(임의 추측 금지 — 모르면 사용자에게 확인).

## 5. CONVENTION / DESIGN / CLAUDE / README (승인형)

- 갱신이 필요한 위치와 추가/수정할 정확한 문구를 **diff 형태로 제시**하고, 승인 후 적용.
- CONVENTION/DESIGN의 새 규칙은 팀 합의 성격이므로 사용자 확인 없이 확정하지 않는다.
- CLAUDE.md 디렉터리 트리·명령어 표는 실제 변경과 어긋난 부분만 최소 수정.

## 6. 마무리

- 적용한 변경과 승인 대기 항목을 요약한다.
- 문서 파일들을 같은 커밋에 포함하도록 안내(`git add` 대상 나열).
- ADR에 `TODO` 마커를 남겼다면 그 사실을 명시해 사용자가 잊지 않게 한다.

## 경계

- 절대 경로·`cn()`·`import type` 같은 코드 규칙은 이 스킬의 대상이 아니다(그건 lint·코드리뷰 영역).
- `.env*` 내용을 문서에 인용하지 않는다.
- `main`에 직접 커밋·push하지 않는다. 작업은 `develop` 기반 브랜치에서.
